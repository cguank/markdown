# 微前端沙盒原理

> 微前端中「沙盒」用于隔离子应用运行环境，避免样式、JS 全局变量、路由等互相污染。本文说明常见沙盒的实现思路与原理。

---

## 一、为什么需要沙盒

在同一个页面里跑多个子应用时，若不隔离会带来：

- **JS 隔离**：多个应用共享 `window`、全局变量、`document`，会互相覆盖（如都改 `window.xxx`、污染原型）。
- **样式隔离**：子应用的 CSS 作用在同一个 document，选择器可能互相影响。
- **路由/历史**：多个应用可能都监听或修改 `location`、`history`，产生冲突。

沙盒的目标：让每个子应用在「自己的环境」里运行，尽量像独立页面一样，同时不破坏主应用和其他子应用。

---

## 二、沙盒的常见类型

| 类型           | 主要手段                     | 隔离程度 | 典型场景     |
|----------------|------------------------------|----------|--------------|
| JS 沙盒        | 代理/快照 window、注入/还原  | 高       | 多实例、多版本 |
| 样式隔离       | Shadow DOM / 前缀 / 卸载恢复 | 中~高    | 多应用同屏   |
| 运行环境隔离   | iframe、Web Worker           | 最高     | 强隔离、第三方 |

下面分别说 **JS 沙盒** 和 **样式隔离** 的原理。

---

## 三、JS 沙盒原理

### 3.1 思路概览

子应用代码里访问的「全局」实际不是真正的 `window`，而是一个**代理对象**或**快照环境**：

- 读：从当前子应用自己的「全局」里取（或按规则从主应用透传）。
- 写：只写进当前子应用的「全局」，不污染主应用和其他子应用。
- 子应用卸载时：把对真实 `window` 的修改回滚或清理，恢复主应用环境。

常见两种实现方式：**快照沙盒（Snapshot）** 和 **代理沙盒（Proxy）**。

### 3.2 快照沙盒（Snapshot Sandbox）

**思路**：在子应用挂载前给当前 `window` 打一份快照；子应用运行期间允许它随意改 `window`；卸载时用快照把 `window` 还原，再保存当前 `window` 的「修改」用于下次激活时恢复。

**简化流程**：

1. **激活**：  
   - 保存当前 `window` 的键值 → `snapshot`（快照）。  
   - 若存在上次的 `modifications`，把这些修改重新应用到 `window`（恢复该子应用上次的全局状态）。
2. **运行**：子应用正常跑，直接改的就是真实 `window`。
3. **卸载**：  
   - 对比当前 `window` 和 `snapshot`，把**多出来的或改了的**记到 `modifications`。  
   - 用 `snapshot` 把 `window` 全部还原（删除新增、恢复旧值）。

**特点**：

- 实现简单，兼容性好（不依赖 Proxy）。
- 同一时间只能有一个快照沙盒生效，否则会互相覆盖 `window`，适合单实例场景。
- 需要遍历 `window` 做 diff，属性极多时会有一定开销。

**伪代码示意**：

```js
class SnapshotSandbox {
  constructor() {
    this.snapshot = {};
    this.modifications = {};
  }

  activate() {
    // 1. 保存当前 window 快照
    for (const key in window) {
      this.snapshot[key] = window[key];
    }
    // 2. 恢复该子应用之前的修改
    Object.keys(this.modifications).forEach(key => {
      window[key] = this.modifications[key];
    });
  }

  deactivate() {
    for (const key in window) {
      if (window[key] !== this.snapshot[key]) {
        this.modifications[key] = window[key]; // 记下修改
        window[key] = this.snapshot[key];       // 还原
      }
    }
    for (const key in this.modifications) {
      if (!(key in this.snapshot)) {
        delete window[key]; // 删除子应用新增的
      }
    }
  }
}
```

**两个子应用修改了同一个 key 会出错吗？**

分两种情况：

- **串行切换（同一时刻只激活一个子应用）**：不会出错。每个子应用各自维护一份 `snapshot` 和 `modifications`。切走时（deactivate）把当前对 `window` 的改动记到自己的 `modifications` 并还原 `window`；切回来时（activate）先按自己的 `snapshot` 还原，再把自己的 `modifications`  applied 回去。所以同一个 key（如 `window.xxx`）在 A 激活时是 A 存的值，在 B 激活时是 B 存的值，各用各的，不会串。
- **同时运行两个子应用**：快照沙盒本身是**单实例**的，同一时间只允许一个快照沙盒生效。若两个子应用同时“挂”在同一个真实 `window` 上，后激活的会把前一个的修改一起拍进自己的快照，切换时谁还原、谁应用会乱，同一个 key 就会互相覆盖、表现异常。这种场景快照沙盒不支持，应改用**代理沙盒**（每个子应用一个 proxy，互不共享同一个 `window`）。

结论：**同 key 在快照沙盒下，只要保证同一时刻只激活一个子应用，就不会出错；若要多子应用同时挂载，应用代理沙盒。**

### 3.3 代理沙盒（Proxy Sandbox）

**思路**：子应用不直接访问 `window`，而是访问一个用 **Proxy** 包过的「假 window」。读可以按规则从假 window、真实 window 或白名单取；写只写入假 window（或一份拷贝），不写真实 `window`。这样多子应用可以同时存在多个沙盒，互不干扰。

**简化流程**：

1. 创建一个「空对象」或 `window` 的拷贝作为子应用的 `fakeWindow`。
2. 用 `new Proxy(fakeWindow, handlers)` 做读写拦截：
   - **读**：优先从 `fakeWindow` 读；若没有且在白名单内，可从真实 `window` 读（如 `document`、`location`、`history` 等需要与主应用一致或受控的）。
   - **写**：只写入 `fakeWindow`，不写入真实 `window`。
3. 子应用执行前，通过「把全局变量指向 proxy」的方式注入（例如用 IIFE 包一层，把 `window` 替换成 proxy，或通过 `with(proxy)` 等），这样子应用里的 `window.xxx` 实际都走 proxy。
4. 卸载时丢弃该 `fakeWindow` 即可，无需还原真实 `window`。

**特点**：

- 多实例友好，每个子应用一个 proxy 实例即可。
- 依赖 `Proxy`，旧浏览器要降级或不用。
- 子应用若用 `var`/未声明直接赋值，可能仍会挂到真实 `window`（取决于加载方式），通常要配合构建或运行时把代码包在「指定全局」的环境里。

**伪代码示意**：

```js
function createProxySandbox() {
  const rawWindow = window;
  const fakeWindow = {};
  const sandbox = new Proxy(fakeWindow, {
    get(target, key) {
      if (key in target) return target[key];
      // 白名单：从真实 window 读
      if (['document', 'location', 'history', 'localStorage'].includes(key)) {
        return rawWindow[key];
      }
      return rawWindow[key]; // 或按需只透传部分
    },
    set(target, key, value) {
      target[key] = value;  // 只写入 fakeWindow
      return true;
    }
  });
  return sandbox;
}
```

实际框架（如 qiankun）还会处理：`document`、`location` 的代理、历史记录同步、子应用 `createElement` 的容器归属等，避免子应用误操作主应用 DOM。

## 四、样式隔离原理

### 4.1 思路概览

- **Shadow DOM**：把子应用的根节点包进一棵 Shadow Tree，子应用样式只作用在 Shadow 内部，不泄露到外部，外部样式也可选择是否进入（`all: initial` 等）。
- **Scoped 前缀**：构建或运行时给子应用选择器加唯一前缀（如 `data-qiankun-xxx`），减少和主应用/其他子应用冲突。
- **卸载时移除**：子应用卸载时把其注入的 `<style>`/`<link>` 移掉，避免残留样式。

### 4.2 Shadow DOM

- 子应用挂载到一个容器节点后，给该容器 `attachShadow({ mode: 'open' })`，子应用的 DOM 和样式都放在这颗 Shadow Root 下。
- 样式只在 Shadow 内生效，实现强隔离；但子应用若依赖「全局样式」或第三方库写死 `document`，需要额外处理。

### 4.3 选择器前缀（Scoped）

- 编译或运行时为子应用根节点加唯一属性，如 `data-qiankun-app="appName"`，并把子应用样式表里的选择器都加上 `[data-qiankun-app="appName"]` 前缀，使样式只作用于该根下。
- 隔离强度弱于 Shadow DOM，但兼容性和实现成本更友好。

## 五、iframe 如何实现沙盒

iframe 的沙盒**不是**你在 JS 里写出来的，而是**浏览器原生**给的：每个 iframe 拥有独立的运行环境，由浏览器保证隔离。

### 5.1 浏览器提供了什么

- **独立的 window**：iframe 内代码里的 `window` 指向 iframe 的 window，不是父页面的。全局变量、`var`、挂到 `window` 上的属性都只在这个 window 里，和父页面、其他 iframe 互不影响。
- **独立的 document**：iframe 有自己的 document、DOM 树、Cookie（按同源策略）、localStorage/sessionStorage（按源隔离）。子应用改 DOM、插样式只影响 iframe 内的 document。
- **独立的 JS 执行环境**：有独立的全局作用域、事件循环（和主页面共享同一个 event loop，但执行栈/全局对象分离）。脚本报错、内存不会直接拖垮主页面（除非通过引用或 postMessage 等显式传过去）。
- **源（origin）与安全策略**：  
  - **同源**：父和子 origin 相同（协议+域名+端口一致）时，父可拿到 `iframe.contentWindow` / `iframe.contentDocument`，子可 `window.parent`，可互相访问 DOM 和部分 API。  
  - **跨源**：不同源时，浏览器会阻止父子直接访问对方 DOM 和大部分对象，只能通过 `postMessage` 通信。这就是「沙盒」：靠**同源策略**把不同源的 iframe 关在各自环境里。

所以 iframe 的沙盒 = **多 document + 多 window + 按源隔离**，由浏览器实现，不需要自己写 Proxy 或快照。

### 5.2 和「沙盒」的对比

| 方面         | 快照/代理沙盒                     | iframe 沙盒                          |
|--------------|------------------------------------|--------------------------------------|
| 谁实现       | 业务/框架用 JS 模拟                | 浏览器原生                            |
| 全局对象     | 共享一个 window，读写被代理/还原   | 每个 iframe 一个 window，物理隔离    |
| DOM / 样式   | 同一 document，靠 Shadow/前缀隔离  | 每个 iframe 一个 document，天然隔离  |
| 是否多 document | 否，单 document                  | 是，多 document                       |
| 通信         | 直接访问同一 window                | 跨源需 postMessage、同源可直接引用   |

### 5.3 小结

iframe 实现沙盒的方式是：**为子页面提供独立的 window、document 和源**，由浏览器保证不同 iframe（及与父页面）之间的隔离；同源可互访、跨源只能通过 postMessage 等 API 通信。没有额外的「沙盒代码」，隔离来自浏览器内核本身。

## 六、和 iframe 的对比

| 维度       | 沙盒（Proxy/快照）     | iframe                    |
|------------|------------------------|----------------------------|
| 隔离强度   | 高（需自己实现完整）   | 最高（独立 document/全局） |
| 性能/通信  | 同文档，无跨域问题     | 有跨域、postMessage 等     |
| 体验       | 路由、主题易统一       | 嵌套、路由、高度同步较麻烦 |
| 适用       | 主应用可控的子应用     | 强隔离、第三方、老系统     |

沙盒是在「同一 document」下用代理/快照模拟隔离；iframe 是真正的多 document 隔离。

---

## 七、小结

- **JS 沙盒**：通过快照（单实例、还原 window）或 Proxy（多实例、假 window）让子应用读写「自己的全局」，不污染主应用。
- **样式隔离**：通过 Shadow DOM 或 Scoped 前缀（或卸载时移除样式）限制子应用 CSS 作用范围。
- **iframe 沙盒**：由浏览器原生提供，每个 iframe 拥有独立的 window、document 和源，靠多 document + 同源策略实现隔离，无需自己实现代理/快照。
- 实际框架（如 qiankun、micro-app）会同时做：JS 代理/快照、样式隔离、子应用生命周期、资源加载与卸载，共同构成完整的「微前端沙盒」。
